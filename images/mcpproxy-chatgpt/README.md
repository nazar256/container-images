# mcpproxy-chatgpt image

Custom MCPProxy.app image for a narrow ChatGPT-facing MCPProxy instance.

It is meant for a separate deployment from the broad local/coding-agent MCPProxy instance. Start with Nextcloud, then optionally add the existing `chrome-devtools-mcp-proxy` as a disabled remote upstream.

## Includes

- MCPProxy.app `mcpproxy` v0.33.5 from the official release tarball.
- Node.js 22 with `npm`/`npx`.
- Python 3.
- `uv`/`uvx`.
- `tini`.
- `procps` for lifecycle checks.
- Non-root `mcpproxy` user.

Persistent paths:

- `/var/lib/mcpproxy` for MCPProxy data and default config.
- `/etc/mcpproxy` for mounted config files.
- `/usr/local/lib/mcpproxy/wrappers` for wrapper scripts.

## Entrypoint defaults

The default command is equivalent to:

```bash
mcpproxy serve \
  --listen "${MCPPROXY_LISTEN:-0.0.0.0:8080}" \
  --data-dir "${MCPPROXY_DATA_DIR:-/var/lib/mcpproxy}" \
  --require-mcp-auth=true \
  --enable-socket=false
```

`MCPPROXY_API_KEY` or `MCPPROXY_API_KEY_FILE` is required by default so the container does not silently start with only an auto-generated admin key. For private first-run setup only, set `MCPPROXY_ALLOW_AUTOGENERATED_API_KEY=true` and read the generated key from logs.

## Secret handling

Prefer Docker Swarm secret files:

```bash
MCPPROXY_API_KEY_FILE=/run/secrets/mcpproxy_admin_api_key
NEXTCLOUD_URL_FILE=/run/secrets/nextcloud_url
NEXTCLOUD_USERNAME_FILE=/run/secrets/nextcloud_username
NEXTCLOUD_PASSWORD_FILE=/run/secrets/nextcloud_password
# or NEXTCLOUD_TOKEN_FILE=/run/secrets/nextcloud_token
```

The Nextcloud wrapper reads `*_FILE` values, exports the corresponding environment variables for the child STDIO MCP process, and never echoes secrets. This keeps raw backend secrets out of MCPProxy upstream `server.Env`; secrets still exist inside the child process environment within the container trust boundary.

## Nextcloud wrapper

Wrapper path:

```text
/usr/local/lib/mcpproxy/wrappers/run-nextcloud-mcp
```

The exact Nextcloud MCP package/command must be supplied at runtime because this image does not invent or install a Nextcloud MCP server package. Set:

```bash
NEXTCLOUD_MCP_COMMAND_JSON='["npx","-y","<actual-nextcloud-mcp-package>"]'
```

or, for a uvx-based server:

```bash
NEXTCLOUD_MCP_COMMAND_JSON='["uvx","<actual-nextcloud-mcp-package>"]'
```

Then configure the upstream as STDIO command `/usr/local/lib/mcpproxy/wrappers/run-nextcloud-mcp` with no raw secrets in `env`.

## Browser upstream example

Do not run `chrome-devtools-mcp` directly in this image initially. Add the existing browser proxy as a disabled remote upstream:

```json
{
  "name": "chrome",
  "url": "http://chrome-devtools-mcp-proxy:8000/mcp",
  "protocol": "http",
  "enabled": false,
  "headers": { "X-API-Key": "<internal-browser-proxy-api-key>" }
}
```

Browser CDP tokens stay in `chrome-devtools-mcp-proxy`, not in this MCPProxy.app container.

## ChatGPT-facing deployment notes

- Public Traefik route should expose only `/mcp/call` over HTTPS.
- Do not publicly expose `/`, `/ui/`, `/api/`, `/swagger/`, `/debug/pprof/`, or `/mcp/all`.
- Keep the UI/admin route private-network-only.
- Keep `require_mcp_auth=true` for internet-facing MCP.
- Use scoped agent tokens for ChatGPT.
- Do not mount the Docker socket unless explicitly choosing that security tradeoff.

## Runtime validation

After repeated ChatGPT-like sessions/reconnects:

```bash
ps -eo pid,ppid,rss,etimes,args \
  | grep -E 'mcpproxy|node|npm|npx|uvx|python|nextcloud|chrome-devtools' \
  | grep -v grep
```

Expected: one steady MCPProxy process plus one steady process tree per enabled STDIO upstream. No one-process-per-session or one-process-per-tool-call growth.
